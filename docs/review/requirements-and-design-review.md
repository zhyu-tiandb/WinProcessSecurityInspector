## 2. 修订后新增的发现

### 2.1 📌 `ProcessInfo` 结构体缺少 `startTime` 字段

| 条目 | 内容 |
|------|------|
| **位置** | 概设 §3 — `ProcessInfo` 结构体定义 |
| **问题** | SRS §6.1 要求采集 "Process Start Time"，概设 `ProcessInfo` 结构体中缺少该字段 |
| **作用** | 启动时间用于：进程树排序、父子关系中的父 PID 复用判断（SRS §6.13）、浏览器同实例识别（SRS §6.14 提及"创建时间"作为识别依据之一） |
| **影响** | 低 —— 字段可在详细设计中补充，不影响当前概设的结构合理性 |
| **建议** | 在 `ProcessInfo` 中补充 `startTime` 字段，建议类型 `std::chrono::system_clock::time_point` 或 `FILETIME`，具体类型由详细设计确定 |

### 2.2 ⚠️ 验收标准中 Window Station / Desktop 的位置说明

| 条目 | 内容 |
|------|------|
| **位置** | SRS §18.1 基础采集验收 |
| **问题** | §18.1 将 Window Station 和 Desktop 列为基础采集验收项，但 Planner §19.1 和概设 §14 将其放在 Phase 3。同一文档内验收条件与阶段计划不直接矛盾（§18 可理解为全功能目标），但读者可能在评审时认为基础验收应包含 WinSta/Desktop |
| **影响** | 低 —— 如果读者理解 §18.1 为最终产品的完整验收基准则无问题 |
| **建议** | 可选：在 §18.1 增加脚注"Window Station 和 Desktop 采集在第三阶段交付，对应详见 §19.3" |

## 3. 整体评价

### 3.1 修订质量

- **响应度**：全部 14 项问题均得到认真对待，每项都有对应的修改
- **精准度**：修改内容针对问题本身，未引入不必要的范围蔓延
- **结构性改善显著**：概设新增 **核心数据结构（§3）、并发与线程模型（§8）、Windows API 依赖与 COM 初始化（§9）、测试策略（§10）、日志策略（§11）** 五个章节，文档完整性大幅提升
- **细节到位**：敏感命令行参数脱敏规则、诊断规则合并策略、API 调用超时约束等均给出了明确约定

### 3.3 进入详细设计的建议

**基于文档当前状态，建议进入详细设计阶段。** 以下提示供详细设计参考：

1. **`ProcessInfo.startTime`** 在详细设计时补充具体类型
2. **`ProcessSecurityContext`** 可考虑增加 `BrowserProcessContext` 字段，或保持当前 `DiagnosisContext` 拆开的设计——拆开更灵活（浏览器诊断只填浏览器字段时不污染普通进程诊断），保持即可
3. **COM Apartment 检测**：概设 §9 提到应检测 `RPC_E_CHANGED_MODE` 并在专用线程执行 —— 建议详细设计中给出具体实现模式（如 COM Worker 线程池）
4. **诊断规则配置**：建议在详细设计中给出 `config/diagnosis-rules.json` 的具体 JSON Schema
